## Why v2

### Market Reality

v1 positioned as "AI-driven chip design automation" with natural language to RTL as the core feature. Market research (April 2026) revealed:

- NL-to-RTL is the most overhyped area in AI EDA. No commercial tool does this reliably. Works for toy examples, fails on anything complex.
- Verification is the real pain point: 40-70% of total chip design effort, 40% of that spent on debugging.
- Taiwan semiconductor talent gap: 34,000 unfilled positions per month, 0.45 applicants per DV opening.
- ChipAgents raised $74M for AI verification. Moores Lab, Bronco AI also in the space. This is where the money flows.
- Open-source EDA (Verilator, Yosys, OpenROAD) is production-ready at 130nm. Hundreds of real tapeouts on SkyWater 130nm.

### Strategic Shift

From: "AI generates chips automatically"
To: "AI assists chip verification learning and workflow automation"

### v1 to v2 Migration

**Keep:**
- Next.js frontend framework, Tailwind CSS, shadcn/ui component library
- Verilator and Yosys integration (Docker sandbox)
- FastAPI backend structure
- i18n system (English + Traditional Chinese)
- Workspace and history management

**Rewrite:**
- Backend architecture: Dragons → Pipeline runner with step functions
- LLM integration: hardcoded vLLM → pluggable LLM provider interface
- Frontend: separate page-per-feature → pipeline visualization as core UI

**Remove as primary feature:**
- NL-to-RTL as the main selling point (keep as optional "Design" step in pipeline)

**Retain as optional:**
- RTL generation from natural language (useful for education, generating starting points)

## Core Features (Priority Order)

### F1: Test Plan Auto-Generation

AI analyzes RTL module and produces a structured verification plan:
- Input/output port analysis
- Functional scenario enumeration
- Edge case and boundary condition identification
- Coverage goal recommendations
- Reset, clock domain crossing, and protocol-specific test scenarios

**Why first:** Highest leverage. Teaches new engineers "what to test" (verification thinking), not just "how to write code." Industry DV seniors spend significant time reviewing and creating test plans.

**AI error tolerance:** High. Test plan is a document, not executable code. Wrong items are easily spotted and removed by the user.

### F2: Testbench Auto-Generation + Coverage-Driven Iteration

Given RTL + test plan, generate SystemVerilog testbenches:
- Generate initial testbench covering test plan scenarios
- Run Verilator simulation automatically
- Analyze real coverage metrics (line, toggle, branch via `--coverage`)
- Identify uncovered paths
- Re-generate additional test cases targeting coverage gaps
- Iterate until coverage target met or max iterations reached

**Closed-loop verification:**
```
Generate Testbench → Compile (Verilator) → Simulate → Coverage Report
       ↑                                                    |
       └──── Coverage < Target? ────── Re-generate ←────────┘
```

**AI error tolerance:** High. Verilator catches syntax errors at compile time. Simulation catches runtime errors. Coverage metrics quantify confidence. Failed generation triggers automatic retry with error feedback.

### F3: Interactive Debug Assistant

When simulation fails, AI explains the failure:
- Parse Verilator error messages and simulation logs
- Identify failing assertions or mismatches
- Explain root cause in plain language (educational value)
- Suggest fixes with code examples
- Highlight suspicious signals in waveform context

**AI error tolerance:** Moderate. Debug suggestions are advisory. Wrong suggestions waste time but don't cause harm. The educational value (explaining what went wrong) is valuable even if the suggested fix is imperfect.

### F4: RTL-to-GDSII Flow Integration

Integrate open-source EDA tools into the pipeline:
- Yosys: RTL synthesis to gate-level netlist
- OpenROAD: Placement and routing
- Magic: DRC/LVS (stretch goal)
- Target: SkyWater 130nm, GF 180nm (open PDKs)

**Why included in MVP:** For education, seeing the full journey from code to physical layout is transformative. Students understand that Verilog becomes real silicon.

### F5: SVA Assertion Generation + Formal Proof (Post-MVP)

- AI generates SystemVerilog Assertions from RTL analysis
- SymbiYosys runs formal verification (bounded model checking)
- Report: proven properties, counterexamples, inconclusive
- Lowest competition, strongest moat, best AI error tolerance (formal tools catch wrong assertions mathematically)

## Technical Architecture

### Target Architecture: Pipeline Engine + Plugin Nodes (C+D Hybrid)

This is the long-term target architecture. Implementation is phased to avoid over-engineering.

```
Pipeline Engine (core, open source, MIT)
  |
  ├── Node: RTL Input
  |     └── Plugin: File Parser (built-in)
  |
  ├── Node: Lint
  |     └── Plugin: Verilator Lint (open source)
  |
  ├── Node: Test Plan Generation
  |     └── Plugin: LLM (pluggable: Claude / GPT / DeepSeek / Qwen)
  |
  ├── Node: Testbench Generation
  |     └── Plugin: LLM (pluggable)
  |
  ├── Node: Simulation
  |     └── Plugin: Verilator Sim (open source)
  |
  ├── Node: Coverage Analysis
  |     └── Plugin: Verilator Coverage (open source)
  |     └── Plugin: Advanced Coverage Analyzer (paid)
  |
  ├── Node: Debug Assistant
  |     └── Plugin: LLM + Log Parser (open source core)
  |     └── Plugin: Waveform Analyzer (paid)
  |
  ├── Node: Synthesis
  |     └── Plugin: Yosys (open source)
  |
  └── Node: Place & Route
        └── Plugin: OpenROAD (open source)

Iteration Loops:
  Coverage < target → back to Test Plan → regenerate → re-simulate
  Lint failure → back to LLM → fix with error context → re-lint
  Sim failure → Debug Assistant → suggest fix → user edits → re-simulate
```

### Implementation Phases

**v2.0 (Month 1-4): Simple sequential runner, no plugin system**

```python
# Start simple. Each step is a function. No abstractions until needed.
async def run_pipeline(rtl: str, config: PipelineConfig) -> PipelineResult:
    lint_result = await verilator_lint(rtl)
    if not lint_result.passed:
        return PipelineResult(failed_at="lint", error=lint_result.errors)

    test_plan = await generate_test_plan(rtl, lint_result, config.llm)
    testbench = await generate_testbench(rtl, test_plan, config.llm)

    for iteration in range(config.max_iterations):
        sim_result = await verilator_simulate(rtl, testbench)
        coverage = await analyze_coverage(sim_result)

        if coverage.score >= config.coverage_target:
            break
        testbench = await improve_testbench(rtl, testbench, coverage, config.llm)

    return PipelineResult(
        lint=lint_result,
        test_plan=test_plan,
        testbench=testbench,
        simulation=sim_result,
        coverage=coverage,
        iterations=iteration + 1,
    )
```

**v2.5 (Month 5-8): Refactor to plugin architecture when complexity demands it**

Only add the plugin system when:
- There are 5+ pipeline steps
- Users request custom steps or tool swaps
- Third-party contributors want to add integrations

### Pipeline Engine Design (Target, v2.5+)

- DAG-based execution engine
- Each node: input schema, output schema, execute(), validate()
- Nodes can be skipped, rerun, or branched
- Pipeline state persisted (resume after interruption)
- Event-driven: nodes emit events for UI updates

### Plugin Interface (Target, v2.5+)

```python
class PipelinePlugin(ABC):
    """Base interface for all pipeline plugins."""

    @abstractmethod
    def name(self) -> str: ...

    @abstractmethod
    def version(self) -> str: ...

    @abstractmethod
    def input_schema(self) -> dict: ...

    @abstractmethod
    def output_schema(self) -> dict: ...

    @abstractmethod
    async def execute(self, input_data: dict, context: PipelineContext) -> dict: ...

    @abstractmethod
    async def validate(self, input_data: dict) -> ValidationResult: ...
```

### LLM Strategy

**Pluggable LLM backend:**
- Tool layer (Verilator, Yosys, OpenROAD): runs locally, no LLM needed
- AI layer (Test Plan, Testbench, Debug, SVA): pluggable LLM provider
- Supported providers: OpenAI API, Anthropic API, self-hosted vLLM, Ollama
- Customer chooses based on IP sensitivity and budget

**Fine-tuned models (long-term differentiation):**
- Fine-tune DeepSeek Coder or Qwen Coder on Verilog/SystemVerilog corpus
- Training data: open-source HDL from GitHub, OpenCores, academic repos
- Research shows 6-25% improvement over general-purpose LLMs on Verilog tasks
- Offer as premium plugin for Managed Service customers

## Frontend Architecture

### Pipeline Visualization (Core UI)

Main screen is an interactive pipeline DAG:
- Each node shows real-time status: pending / running / success / warning / failed
- Click into any node to see detailed results
- Iteration loops visually animated (coverage feedback arrow)
- Progress percentage and elapsed time per node

### Education Mode vs Professional Mode

**Education mode (default for Tier 1):**
- Step-by-step explanations at each pipeline node ("Why do we lint first?")
- AI reasoning visible (why these test cases were chosen)
- Guided tutorials and hints
- Slower, more verbose output

**Professional mode (for Tier 2/3):**
- Minimal explanations, focus on results
- Batch execution, no pauses between steps
- Export results in standard formats (coverage reports, VCD files)
- CLI/API first, Web UI optional

### Node Detail Views

- **Test Plan:** Structured list of scenarios with checkboxes, AI reasoning
- **Testbench:** Code editor with syntax highlighting, diff view for iterations
- **Simulation:** Pass/fail summary, assertion results, log output
- **Coverage:** Line/toggle/branch coverage with heatmap on source code
- **Debug:** Plain-language explanation, suspicious signal highlights
- **Synthesis:** Gate count, area, timing summary, netlist viewer
- **P&R:** Layout visualization (stretch goal)

### Tech Stack (unchanged from v1)

- Next.js (App Router)
- TypeScript
- Tailwind CSS
- shadcn/ui + Geist

## Business Model

### Open Core (MIT License)

**Free (open source):**
- Pipeline Engine
- All basic plugins (Verilator, Yosys, OpenROAD integration)
- LLM integration layer (bring your own API key)
- Web UI with pipeline visualization
- CLI and API access

**Paid plugins:**
- Fine-tuned Verilog/SV models
- Advanced coverage analysis and reporting
- Waveform-based debug assistant
- Enterprise SSO and audit logging
- Priority support

### Managed Service

- On-premise deployment for IC companies
- Deployment fee + annual maintenance contract
- Includes fine-tuned models and all paid plugins
- IP never leaves customer's network
- Target pricing: NT$500K-2M/year per team

### Early Revenue Options (Before Managed Service)

- **LLM API relay:** Users use XylonStudio's API key, pay a markup per token. Simple to implement, immediate revenue potential.
- **SaaS Pro tier:** Free tier with 50 pipeline runs/month, Pro at $29/month for unlimited runs + advanced coverage reports.
- **Consulting/workshops:** Paid verification methodology workshops for IC companies, using XylonStudio as the teaching tool.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLM-generated testbenches too unreliable | Core value prop fails | Coverage-driven iteration loop catches errors automatically. Verilator is the safety net. Start with simple designs, expand complexity gradually. |
| Professors don't adopt (wrong market assumption) | No Tier 1 users | Phase 0 validation. If professors say no, pivot to FPGA community (Tier 3) as Day 1 target. |
| IC companies won't share RTL with cloud LLM | Lose Tier 2 market | On-premise deployment + self-hosted LLM option. IP never leaves network. |
| ChipAgents / Moores Lab dominate verification market | Competition | Different market (education-first, open source) and different approach (pipeline vs. point solution). They target enterprise, we target learning + small teams. |
| Fine-tuning requires too much training data | Differentiation delayed | Start with general-purpose LLMs (Claude/GPT). Fine-tuning is Phase 2, not MVP blocker. |
| 6-8 month timeline too aggressive | Delayed launch | v2.0 (Month 1-4) is a shippable product on its own. Debug + Synthesis (v2.5) can be deferred. |
| Open-source EDA tool limitations (SymbiYosys SVA support) | F5 constrained | Start with supported SVA subset. Add Tabby CAD integration later. F5 is post-MVP anyway. |
| Zero revenue for 8-12 months | Cash flow gap | LLM API relay and SaaS Pro tier can generate early revenue. Consider seed funding if needed. |
| Over-engineering pipeline/plugin architecture | Wasted time | Start with sequential runner. Only refactor to plugins when complexity demands it. |
